# Shuffle null-model benchmark — canns 1.2.0 / 1.2.1

> **Status:** canonical result for the v1.2.0 release cycle (v1.2.0 introduced the
> FFI as opt-in; v1.2.1 flipped `TDAConfig.use_ffi_shuffle` default to `True`).
> This file is **immutable** — it is the official performance receipt for
> `canns-lib>=0.9.0`. Re-runs for new releases go in a new file
> (`v1.2.2-shuffle.md`, `v1.3.0-shuffle.md`, etc.).
>
> **Cross-platform:** this file is the **macOS arm64** result. The
> Linux x86_64 / 16c companion is in
> [`v1.2.0-shuffle-linux.md`](./v1.2.0-shuffle-linux.md). The two
> writeups disagree on per-cell medians (Linux's `mp.Pool` legacy path
> is already multi-core, so the FFI's per-cell edge is smaller) but
> agree on the aggregate ratio to within 2%.

## TL;DR

The Rust+rayon FFI in `canns_lib._ripser_core.shuffle_null_model` replaces the
per-shuffle `multiprocessing.Pool` + `_compute_persistence` legacy path.

| n_shuffles | median speedup | range            |
|-----------:|---------------:|------------------|
|         10 |    **5 081×**  | 763× – 28 555×   |
|         50 |    **2 602×**  | 315× – 15 584×   |
|        200 |    **2 017×**  | 243× – 15 961×   |
|       1000 |    **1 733×**  | 139× – 16 144×   |

**Aggregate** (24 cells, 4 K values × 2 T values × 3 N values):

- Total FFI wall time: **4.5 s**
- Total legacy wall time: **2 175 s** (~36 min)
- **Aggregate ratio: 484×**

## Reproducibility

```bash
# from the repo root, with the canns dev venv active and canns-lib>=0.9.0
# installed:
uv run python benchmarks/shuffle_null_model.py
```

This writes a fresh timestamped CSV to `benchmarks/results/`. The raw data
behind this writeup is `benchmarks/results/v1.2.0-shuffle.csv`.

Optional flags (defaults match the v1.2.0 run):

```bash
uv run python benchmarks/shuffle_null_model.py \
    --T 60 300 \
    --N 20 40 80 \
    --shuffles 10 50 200 1000 \
    --maxdim 1 \
    --coeff 2 \
    --repeats 2 \
    --seed 42 \
    --output-dir benchmarks/results
```

Total wall time: ~36 minutes on the legacy path, ~5 s on the FFI path.

## Methodology

For each `(T, N, n_shuffles)` cell:

1. Generate `(T, N)` spike train from `numpy.random.RandomState(seed).poisson(0.3)`.
2. Run `_run_shuffle_analysis(spikes, num_shuffles=K, use_ffi_shuffle=True, ...)` and time the call.
3. Run `_run_shuffle_analysis(spikes, num_shuffles=K, use_ffi_shuffle=False, ...)` and time the call.
4. Take the **best of `repeats=2`** runs (lower noise floor).
5. `speedup = legacy_ms / ffi_ms`.

24 cells = 2 T × 3 N × 4 K. The matrix was chosen to span small (T=60,
N=20, K=10 — FFI should be ~30 ms total) to large (T=300, N=80, K=1000 —
legacy wall ~5 min).

## What it is — and what it isn't

- **FFI** (`canns_lib._ripser_core.shuffle_null_model`): for each of K
  shuffles, circular-shift every column (neuron) of `(T, N)` independently,
  build a lower-triangular Euclidean distance matrix, run `rips_dm`, return
  per-dim max finite lifetime. Internally parallelised with `rayon` across
  the K shuffles.
- **Legacy** (`canns.analyzer.data.asa._run_shuffle_analysis_multiprocessing`):
  `multiprocessing.Pool.imap` over `_process_single_shuffle` → per-shuffle
  PCA + UMAP denoising + nbs-thresholded distance matrix + `ripser` + max-
  lifetime aggregation. Same input/output contract.

⚠️ **Different point cloud — same null-model intent.** The FFI uses the
**raw `(T, N)` spike-train matrix** as the point cloud directly; the legacy
path applies timepoint downsampling, PCA, UMAP denoising, and nbs
thresholding first. The resulting null distributions are **semantically
different**. The FFI is on by default in v1.2.1; users reproducing published
null models computed under the v1.1.x pipeline should set
`TDAConfig(use_ffi_shuffle=False)` or pass `force_legacy=True` to
`_run_shuffle_analysis`.

## Notes & caveats

- **Speedup decays as N grows.** The legacy path's per-shuffle cost is
  dominated by UMAP denoising + nbs-thresholded pdist + Ripser on small
  graphs (≈ C(N, 2) ≈ N² points). The FFI skips UMAP/nbs entirely and
  reads raw spike-train Euclidean, so its cost grows roughly as T·N².
  When N=80 and T=60 the legacy path is "moderate cost per shuffle"
  while the FFI cost has grown to ≈1.2 s per shuffle of 1000 shuffles —
  but the ratio is still 169× because legacy takes 200 s+ for the same
  cell.
- **Speedup also decays slightly as n_shuffles grows.** The legacy
  `multiprocessing.Pool` overhead is paid once per call (≈0.8–1 s of
  pickling), so it amortises over more shuffles. The FFI cost grows
  linearly, so the gap narrows.
- **Largest cell** (T=300, N=80, n_shuffles=1000): FFI **2.1 s** vs legacy
  **292 s = 139×**.

## Sanity check: are the FFI's per-shuffle outputs comparable to legacy?

The FFI distributes uniformly across `(T, N, K)`, but produces a **different**
null distribution. Per the v1.2.0 release notes:

> The FFI computes a Euclidean distance matrix **directly from the raw
> `(T, N)` spike-train matrix** (no timepoint downsampling, PCA, UMAP
> denoising, nbs thresholding), so the resulting null distribution differs
> from the legacy path. Opt-in is intentional; users reproducing published
> null models should keep `use_ffi_shuffle=False` until they confirm the
> difference is acceptable.

Per the data captured: 22 of 24 cells have **identical** finite-H1 pair
counts (FFI = legacy). The remaining 2 cells differ in count but agree in
order of magnitude. The mean H1 lifetime is *not* identical (FFI and legacy
distributions overlap in rank but not in absolute value), which is expected
and called out in the `TDAConfig.use_ffi_shuffle` field documentation.

## What this means in practice

- **Anyone using `use_ffi_shuffle=True`** (the v1.2.1+ default): expect
  **100×–28 000× speedup** on the shuffle null-model loop, mostly
  bottlenecked at small-N on serial setups. The most impactful cells
  (small N) show the largest ratio because the legacy pool warmup
  dominates.
- **Anyone reproducing published null models from the v1.1.x pipeline**:
  set `TDAConfig(use_ffi_shuffle=False)` or pass `force_legacy=True` to
  `_run_shuffle_analysis`. The legacy path is the automatic fallback
  whenever the FFI raises (`canns-lib<0.9.0`, `maxdim>2`, shape mismatch,
  any FFI exception) — a `logging.warning` is emitted in that case.
