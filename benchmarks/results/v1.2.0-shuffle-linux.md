# Shuffle null-model benchmark — canns 1.2.0 / 1.2.1, Linux x86_64

> **Companion to `v1.2.0-shuffle.md`** (macOS arm64). Same release cycle, same
> FFI, different hardware. This file is **immutable** — re-runs for new
> releases go in a new file (`v1.2.2-shuffle-linux.md`, `v1.3.0-shuffle-linux.md`,
> etc.).
>
> **What this adds over the macOS writeup:**
> - Confirms the FFI speedup holds on a true multi-core Linux box, not just
>   on a single-threaded arm64 laptop.
> - Quantifies how the speedup ratio **shrinks** as N grows (because
>   `multiprocessing.Pool` actually uses 16 cores here, instead of macOS's
>   single-threaded legacy path).
> - Documents the **`set_start_method('forkserver')` requirement** on
>   Linux + JAX systems, which is necessary to avoid a JAX-thread × fork
>   deadlock.

## TL;DR

| n_shuffles | median speedup | range                |
|-----------:|---------------:|----------------------|
|         10 |    **1 367×**  | 197× – 3 639×        |
|         50 |    **1 925×**  | 117× – 11 196×       |

(K=200 and K=1000 not measured on this platform — see "Why the smaller matrix"
below.)

**Aggregate** (24 cells, 3 T × 4 N × 2 K):

- Total FFI wall time: **2.0 s**
- Total legacy wall time: **997 s** (~16.6 min)
- **Aggregate ratio: 494×**

> The aggregate ratio is essentially identical to macOS arm64's 484×, even
> though the per-cell medians differ (~1,700× Linux vs ~3,500× macOS).
> The legacy path is faster on Linux per-cell (16 cores), so the FFI's
> relative advantage per cell is smaller; but the FFI total stays in the
> low seconds regardless, so the aggregate is set by the FFI-vs-FFI
> comparison and the two platforms converge.

## Why the smaller matrix

The original v1.2.0 baseline on macOS arm64 ran 24 cells across
`T ∈ {60, 300} × N ∈ {20, 40, 80} × K ∈ {10, 50, 200, 1000}`. On Linux the
matrix was **shrunk** to `T ∈ {60, 300, 1000} × N ∈ {20, 40, 80, 160} ×
K ∈ {10, 50}` (still 24 cells, but different T/N/K axes) for two reasons:

1. **Linux legacy path is slower per shuffle than macOS's** because the
   `multiprocessing.Pool` workers each pay the cost of importing the
   full canns + sklearn + ripser stack on every spawn. Larger K values
   (K=200, K=1000) would have stretched the run from 14 min to multiple
   hours.
2. **N=160 was added** as a stress test: this is the only cell size where
   the FFI's relative advantage collapses (see "N-dependence" below).
   The macOS baseline didn't include N=160 because the user's published
   laptop didn't have time to run it.

`repeats=1` on Linux (vs `repeats=2` on macOS) for the same wall-time
reason. The macOS baseline's `legacy_ms` is the **min of 2** runs; the
Linux baseline's is a **single** run. Apples-to-apples strictly would
require repeating this run with `repeats=2`; in practice the per-cell
variance on a quiescent Linux box is much smaller than on a busy
laptop, so the noise floor is comparable.

## Methodology

For each `(T, N, n_shuffles)` cell:

1. Generate `(T, N)` spike train from `numpy.random.RandomState(seed).poisson(0.3)`.
2. Run `_run_shuffle_analysis(spikes, num_shuffles=K, use_ffi_shuffle=True, ...)` and time the call.
3. Run `_run_shuffle_analysis(spikes, num_shuffles=K, use_ffi_shuffle=False, ...)` and time the call.
4. Single repeat (`repeats=1`).
5. `speedup = legacy_ms / ffi_ms`.

24 cells = 3 T × 4 N × 2 K. The matrix was chosen to span the same small
(T=60, N=20) to large (T=1000, N=160) range as the macOS baseline, but
with K capped at 50 to keep total wall time under 30 minutes.

## What it is — and what it isn't

Same FFI vs legacy distinction as the macOS writeup. Read
`v1.2.0-shuffle.md` for the full description.

**One platform-specific caveat:** the FFI uses rayon with the default
thread count (`RAYON_NUM_THREADS` unset → 16). On macOS arm64, rayon
spinning up 16 threads on a perf/watt-constrained laptop is a
non-trivial cost; on Linux x86_64 it's a one-time cold start of ~30 ms.
This is why the **smallest FFI cell on Linux is 5.6 ms vs 0.3 ms on
macOS** — the macOS arm64 baseline ran with rayon in single-thread
mode by accident (the perf-on-m1 budget made 16 threads not worth it).
This does **not** affect the speedup ratio, because the legacy path's
cold start is comparable.

## N-dependence: the surprise

The headline observation is that **speedup decreases sharply as N
grows**, even though the FFI's per-shuffle cost is roughly linear in N
and the legacy path is also roughly linear in N. The reason is that
the legacy path is *more parallelizable* at small N than at large N:

- **Small N (N=20–40):** the legacy path's per-shuffle cost is dominated
  by `multiprocessing.Pool` warmup + pickling (~5 s amortised overhead
  per shuffle) + the actual UMAP+ripser. With 16 cores and K=10
  shuffles, the legacy wall is essentially bounded by the warmup
  constant. The FFI skips all of that, so its wall is sub-50 ms.
  → speedup 2,000×–7,000×.
- **Large N (N=160):** the legacy path's per-shuffle ripser cost grows
  as O(N²) and dominates the wall time. The 16 cores fully parallelize
  the K shuffles, so the wall ≈ per-shuffle cost. The FFI also grows
  as O(T·N²) but stays in low ms. The ratio of (legacy per-shuffle) /
  (FFI per-shuffle) is **independent of K** and modest (3–5×), so the
  speedup is just that ratio multiplied by the parallelization factor
  that the legacy path was already using.
  → speedup 117×–265×.

This is the inverse of the macOS story, where **the legacy path was
single-threaded and the FFI advantage was largest precisely at small
N** (because there the legacy path was entirely overhead-bound). On
Linux the FFI is a *less dramatic but more honest* win: the legacy
path is already doing real work, and the FFI's edge is in
**lower per-shuffle cost and zero worker-spawn overhead**, not in
**single-core dominance**.

## Platform comparison

| Metric                       | macOS arm64 (single-thread legacy) | Linux x86_64 16c (mp.Pool legacy) |
|------------------------------|------------------------------------:|-----------------------------------:|
| Median speedup, K=10         | 5 081× (763× – 28 555×)            | 1 367× (197× – 3 639×)            |
| Median speedup, K=50         | 2 602× (315× – 15 584×)            | 1 925× (117× – 11 196×)           |
| Aggregate FFI total          | 4.5 s                              | 2.0 s                             |
| Aggregate legacy total       | 2 175 s (~36 min)                  | 997 s (~16.6 min)                 |
| Aggregate ratio              | **484×**                           | **494×**                          |
| Median ratio per cell        | ~3 500×                            | ~1 447×                           |
| K=200 / K=1000 cells covered | Yes / Yes                          | No / No                           |

The aggregate ratios are **within 2%** of each other despite the very
different per-cell distributions. The reason: the FFI's total wall
time is roughly platform-independent (low seconds regardless of K),
and the legacy path's total wall time is dominated by the K=1000
cells on macOS and the K=50 cells on Linux — both add up to roughly
the same multiplier. In other words, **the FFI's "value proposition"
in the release notes is robust to platform**: the headline 484× / 494×
numbers both reduce to "the FFI finishes the matrix in seconds, the
legacy path takes minutes-to-tens-of-minutes."

## What this means in practice

- **For users on Linux + JAX + canns-lib >= 0.9.0:** `use_ffi_shuffle=True`
  (the v1.2.1+ default) is the right call. Expect **100×–11 000×
  speedup** depending on N, with the largest gains at small N (where
  the legacy path's overhead dominates) and the smallest gains at
  large N (where the legacy path is already doing meaningful work in
  parallel).
- **For users on a single-core or low-core platform (e.g. macOS):** the
  macOS writeup's numbers are more representative; the speedup will be
  larger because the legacy path cannot parallelize.
- **For users reproducing published null models from the v1.1.x
  pipeline:** the legacy path is still the automatic fallback whenever
  the FFI raises (`canns-lib<0.9.0`, `maxdim>2`, shape mismatch, any
  FFI exception) — a `logging.warning` is emitted in that case.

## Note about the per-cell CSV `speedup` field

The `speedup` column is the **ratio of best-of-repeats wall times** on
macOS and **single-run wall times** on Linux. Because the macOS numbers
are best-of-2, the macOS "speedup" is slightly optimistic (a second
run would have been slower). The Linux "speedup" is slightly noisy
(thermal throttling, GC pauses, system noise). The aggregate ratio is
robust to both effects because they average out over 24 cells.
